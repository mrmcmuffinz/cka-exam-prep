# Assignment 1: Pod Fundamentals

This assignment is the first in a planned series of pod-focused CKA practice assignments. It targets pod construction fundamentals only, deliberately narrow in scope so that you can build deep muscle memory for the performance-based exam format without getting distracted by adjacent topics. Everything in this assignment assumes you have completed sections S1 through S6 of the Mumshad CKA course (Core Concepts, Scheduling, Logging and Monitoring, Application Lifecycle Management, Cluster Maintenance) along with the S7 Security material through KubeConfig. You should have a working `kind` cluster running on rootless nerdctl, with `kubectl` pointed at it.

## File Overview

The assignment is split across four files. `README.md` (this file) gives you the map. `pod-fundamentals-tutorial.md` is a walkthrough that builds and inspects a single multi-container pod from start to finish, introducing each spec field as it comes up and showing both imperative and declarative workflows. `pod-fundamentals-homework.md` contains the 15 progressive exercises you will work through on your own, organized into five difficulty levels of three exercises each. `pod-fundamentals-homework-answers.md` contains complete solutions, diagnostic reasoning for the debugging exercises, a common mistakes section, and a verification cheat sheet.

## Recommended Workflow

Work through the tutorial first, end to end, in your own cluster. Do not skim. The tutorial introduces every concept and every inspection command that the exercises rely on, and it uses a dedicated namespace (`tutorial-pod-fundamentals`) that will not collide with any of the exercise namespaces. Once the tutorial is complete, start the homework from Level 1 and work forward. Do not jump ahead to Level 5 debugging exercises before finishing the earlier levels, because the progression is designed to build the diagnostic instincts you need for the hardest exercises. Use the answer key only after you have genuinely attempted an exercise, and treat debugging exercises as a chance to practice the diagnostic sequence (`kubectl get pod`, `kubectl describe pod`, `kubectl logs`, `kubectl get pod -o yaml`) before reaching for the solution.

## Difficulty Progression

Level 1 exercises are single-concept construction tasks with a narrow focus, meant to verify you can build a pod with one specific property correctly. Level 2 exercises combine two or three concepts in a single pod, forcing you to get multiple fields right simultaneously. Level 3 exercises give you broken pod YAML and ask you to diagnose and fix a single clear issue, with failure modes mixed across the three exercises so that you see both pre-start failures (Pending, ImagePullBackOff, Init failure) and post-start failures (CrashLoopBackOff, wrong output, immediate exit). Level 4 exercises are build tasks that construct realistic multi-container pods with meaningful coordination between containers, typically using init containers and emptyDir volumes. Level 5 exercises combine multiple bugs in a single pod spec or introduce subtle coordination issues that require a deep understanding of pod lifecycle and container interaction to diagnose.

## Prerequisites

You need a running `kind` cluster created with rootless nerdctl (`KIND_EXPERIMENTAL_PROVIDER=nerdctl kind create cluster`), and `kubectl` configured to talk to it. Verify this with `kubectl get nodes` before you start. You also need the CKA course sections S1 through S6 complete, plus the first half of S7 through KubeConfig, because a handful of inspection steps in the tutorial assume you understand kubeconfig contexts and basic cluster architecture. No ConfigMaps, Secrets, Services, Deployments, probes, resource limits, or scheduling primitives are used in this assignment, so do not worry if those topics are not yet solid.

## Estimated Time Commitment

Plan for about 45 to 60 minutes to work through the tutorial attentively. The 15 exercises combined should take roughly three to five hours, with Levels 1 and 2 running fast (maybe 10 minutes each), Level 3 debugging exercises running 10 to 20 minutes each depending on how quickly you spot the issue, Level 4 build exercises running 20 to 30 minutes each, and Level 5 exercises taking the longest at 30 to 45 minutes each. If you time-box exercises the way the CKA exam forces you to, these numbers give you a realistic sense of how much faster you need to get before exam day.

## Scope Boundary and What Comes Next

This assignment covers pod construction fundamentals only. Topics that belong in later assignments are explicitly excluded so that each assignment stays narrow enough to master. The planned sequence is as follows. Assignment 2 will cover Pod Configuration Injection (ConfigMaps and Secrets, as both env vars and volume mounts). Assignment 3 will cover Pod Health and Observability (liveness, readiness, and startup probes, lifecycle hooks, termination grace period tuning). Assignment 4 will cover Pod Scheduling (node selectors, node affinity, taints, tolerations, topology spread constraints). Assignment 5 will cover Pod Resources and QoS (requests, limits, QoS classes, eviction behavior). Assignment 6 will cover Multi-Container Patterns (sidecar, ambassador, adapter, shared process namespace, native sidecars with `restartPolicy: Always` on init containers). Assignment 7 will move up a layer to Workload Controllers (ReplicaSets, Deployments, DaemonSets, StatefulSets).

If you find yourself wanting to use a ConfigMap, a probe, or a nodeSelector while doing these exercises, resist the urge. Those are valuable tools, but they belong to later assignments, and the point of this assignment is to build instinctive fluency with the pod spec itself before layering anything else on top.

## Key Takeaways After Completing This Assignment

By the time you finish all 15 exercises, you should be able to construct a pod from scratch using either `kubectl run` with imperative flags or a YAML manifest written from memory, choose correctly between `command` and `args` knowing how they map to Docker's ENTRYPOINT and CMD, set the right `restartPolicy` for the workload (Always for long-running services, OnFailure for retry-on-error batch jobs, Never for one-shot tasks), pick an appropriate `imagePullPolicy` and understand when the default changes, inject pod metadata into a container using the downward API without needing the documentation in front of you, build multi-container pods with init containers that prepare data on an `emptyDir` for the main containers to consume, and diagnose any pod that is not running correctly by reading `kubectl describe` output (specifically the Events section, Conditions, and container statuses) and `kubectl logs` output including the `--previous` and `-c` variants.
