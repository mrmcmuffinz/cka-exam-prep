# Assignment 1: Admission Controllers and ValidatingAdmissionPolicy

Admission controllers are the third phase of the Kubernetes API server request flow. After authentication identifies the caller and authorization decides whether the action is permitted, admission controllers either mutate the incoming object or validate it before persistence to etcd. This assignment is the only one in its series; it covers the full CKA-relevant surface in one pass. The focus is on the request flow (where admission sits relative to authn and authz), the built-in admission controllers kubeadm enables by default (`NamespaceLifecycle`, `LimitRanger`, `ResourceQuota`, `ServiceAccount`, `DefaultStorageClass`, `MutatingAdmissionWebhook`, `ValidatingAdmissionWebhook`, `PodSecurity`), how to read and modify the enabled set via the `--enable-admission-plugins` and `--disable-admission-plugins` flags on the API server, the `ValidatingAdmissionPolicy` resource (CEL-based, stable since Kubernetes 1.30) as the webhook-free extension mechanism the CKA expects candidates to use, and the diagnostic workflow for admission errors from the learner's kubectl output back to the plugin or policy that rejected the request.

## Recommended Workflow

Work through the tutorial with a running cluster and a terminal handy. The tutorial starts from the request flow conceptually (reading the API server's `--enable-admission-plugins` flag), shows a built-in mutating controller in action (`DefaultStorageClass` adding a missing `storageClassName` to a PVC), shows a built-in validating controller in action (`ResourceQuota` rejecting a pod that exceeds a namespace quota), and then builds one complete `ValidatingAdmissionPolicy` with increasing complexity: a simple image-registry constraint, a replica-count cap, and finally a policy that consumes a `paramKind` for its thresholds. The final step walks the three `validationActions` (`Deny`, `Warn`, `Audit`) so the differences in user-visible behavior are observable. When you move into the homework, Level 1 and Level 2 exercises focus on construction (write the policy from requirements); Level 3 is debugging silent failures in admission policies; Level 4 covers real production patterns (paramKind, multi-validation policies); Level 5 includes an SLA-driven policy design task, a multi-bug debug scenario, and a request-flow tracing exercise. Use the answer key only after a genuine attempt.

## Difficulty Progression

Level 1 practices the core construction of a `ValidatingAdmissionPolicy` and its `ValidatingAdmissionPolicyBinding`: a CEL expression that enforces an image registry prefix, a replica count cap, and a label-required constraint. Level 2 adds nuance: the `Warn` enforcement action (the request succeeds, but kubectl prints a warning), the `Audit` action (the request succeeds silently but is recorded in the audit log), and a multi-validation policy where two CEL expressions must both pass. Level 3 is debugging with the anti-spoiler convention: the three exercises cover a `matchConstraints` that does not match the target (policy has no effect), a binding that references the wrong policy name (binding never fires), and a CEL expression that is syntactically valid but semantically wrong (policy accepts what it should reject). Level 4 builds a `paramKind`-driven policy with a `ValidatingAdmissionPolicyBinding` supplying different parameters for different namespaces, a policy that inspects the `oldObject` to block specific field mutations on update, and a policy that uses the `messageExpression` to produce a dynamic error message. Level 5 is advanced: an SLA-driven design for a cluster guardrail set, a cascading multi-bug admission debug where the request-flow sequence has to be traced, and a request-flow authoring exercise that writes the entire chain of objects (Policy, Binding, ParameterConfigMap) for a production-style requirement.

## Cluster Requirements

A single-node kind cluster is sufficient. Kind's kubeadm-provisioned API server ships with the default admission plugins enabled, including `ValidatingAdmissionPolicy` support (which has been GA since Kubernetes 1.30 and is enabled by default in 1.32+). See `docs/cluster-setup.md#single-node-kind-cluster` for the cluster creation command and verification. No extra components (metrics-server, MetalLB, Calico, Gateway API CRDs) are required.

Recovery discipline matters because some exercises disable or alter the API server's admission plugin set by editing `/etc/kubernetes/manifests/kube-apiserver.yaml`. Every such exercise has an explicit revert step; follow it before moving on so later exercises are not contaminated by the previous fix. If the cluster does become unusable, delete and recreate it per the cluster-setup document.

## Scope Boundary and What Comes Next

This assignment covers the admission phase of the request flow, focusing on built-in admission controllers, the API server's admission plugin configuration, and `ValidatingAdmissionPolicy` with CEL expressions. It does not cover authentication (who you are), which is `exercises/18-tls-and-certificates/`. It does not cover authorization (what you can do), which is `exercises/12-rbac/`. It does not cover Pod Security Admission specifically at depth (though `PodSecurity` is one of the built-in admission controllers listed in the tutorial), which is `exercises/14-14-pod-security/assignment-1`. It does not cover `LimitRange` and `ResourceQuota` resource specs as objects (those are in `exercises/01-01-pods/assignment-5`); the admission enforcement of those resources appears here only as a talking point to illustrate how a built-in admission controller interacts with a Kubernetes API object. Writing a custom admission webhook server in Go or another language is out of CKA scope; the assignment focuses on `ValidatingAdmissionPolicy` because it is the extension mechanism the exam tests and it requires no webhook server.

## Key Takeaways After Completing This Assignment

By the time you finish all 15 exercises, you should be able to place admission in the four-stage API server request flow (authentication, authorization, admission, persistence) without hesitation, and know that admission has two sub-phases (mutating first, validating second). You should know the common built-in admission controllers kubeadm enables by default and recognize at least one failure-mode error message from each (for example, the `ResourceQuota` denial format, the `LimitRanger` default-injection behavior, the `NamespaceLifecycle` rejection when writing into a terminating namespace). You should know where the API server's `--enable-admission-plugins` flag lives on disk (the static pod manifest at `/etc/kubernetes/manifests/kube-apiserver.yaml`) and how the edit propagates via kubelet's static pod reconcile loop. You should be able to write a `ValidatingAdmissionPolicy` from requirements using `apiVersion: admissionregistration.k8s.io/v1`, with `spec.matchConstraints.resourceRules` specifying which resources and operations the policy applies to, one or more `spec.validations[]` entries each containing a CEL `expression`, a `message`, and a `reason`, a `spec.failurePolicy` of `Fail` or `Ignore`, and an optional `spec.paramKind` reference to a parameter resource. You should be able to write the matching `ValidatingAdmissionPolicyBinding` with `spec.policyName`, `spec.validationActions` (some subset of `Deny`, `Warn`, `Audit`), `spec.matchResources` for namespace or resource scoping, and an optional `spec.paramRef`. You should know the three CEL variables most CKA-relevant exercises use (`object` for the incoming resource, `oldObject` for updates, `params` for parameter-driven policies) and be comfortable with the string functions (`startsWith`, `contains`, `matches` for regex) and numeric comparisons that make up most real validation expressions. Finally, you should be able to diagnose an admission problem by reading the exact error message kubectl prints, tying the message back to the specific plugin or policy name it cites, and using `kubectl get validatingadmissionpolicies` and `kubectl describe validatingadmissionpolicybinding` to confirm the expected policies are in force.
