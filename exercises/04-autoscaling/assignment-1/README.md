# Assignment 1: Workload Autoscaling

Kubernetes gives workloads three separate ways to adjust their resource footprint: HorizontalPodAutoscaler (HPA) scales the number of pods up or down based on observed metrics, in-place pod resize changes a running pod's CPU and memory requests without restarting its container, and VerticalPodAutoscaler (VPA) recommends or applies new resource requests to make pods the right size. This is the only assignment in the Autoscaling series; it covers the complete CKA-relevant surface in one pass. The focus is on the HPA (`autoscaling/v2`) spec, metrics-server as the dependency that makes CPU and memory HPAs work at all, HPA behavior configuration for scale-up and scale-down tuning, in-place pod resize (GA in Kubernetes 1.33), VPA at a conceptual level including the three update modes (`Off`, `Initial`, `Auto`), and the diagnostic workflow for every failure mode the CKA tests.

## Recommended Workflow

Work through the tutorial with a live cluster and a running metrics-server. The tutorial builds one complete autoscaling workflow from the metrics-server check through HPA creation, load generation, scale-out observation, scale-down stabilization, behavior tuning, in-place resize, and VPA concepts. Every step is run against a real cluster; watching utilization climb and the HPA react is the part of this topic that cannot be learned from documentation alone. Once the tutorial is done, move through the 15 homework exercises in order. Levels 1 and 2 build the core HPA and in-place resize muscle memory. Level 3 is debugging: three realistic silent failures that produce `<unknown>` in the TARGETS column or `unable to get metrics` in the HPA events. Level 4 covers multi-metric HPAs, custom behavior tuning, and combining HPA with in-place resize. Level 5 includes an SLA-driven HPA design task, an advanced flapping-HPA debug scenario, and a VPA-concepts authoring exercise. Use the answer key only after a genuine attempt, and read the Diagnosis section first when tackling debug exercises.

## Difficulty Progression

Level 1 practices basic HPA construction: a CPU-based HPA targeting a Deployment, a memory-based HPA, and an HPA with custom stabilization behavior. Level 2 combines concepts: HPA targeting a StatefulSet (HPA can scale any workload controller with a `scale` subresource), in-place resize of a Pod's CPU request without restarting the container, and in-place resize with `RestartContainer` policy where the restart is required. Level 3 is debugging with the anti-spoiler convention applied: exercise headings are bare, objectives state the desired end state, each exercise contains one silently-failing problem. The three scenarios cover the three most common real HPA failures: the target workload has no `resources.requests`, metrics-server is missing or unhealthy, and the `scaleTargetRef` points at a workload that does not match the Deployment's name or kind. Level 4 moves to production-realistic tasks: multi-metric HPA balancing CPU and memory, asymmetric scale-up and scale-down behavior, and the combination of HPA plus in-place resize where the controller both horizontally grows the pod set and vertically grows the largest pod. Level 5 is advanced: an SLA-driven HPA design task with specific scale-up latency and scale-down smoothness targets, a debugging scenario where an HPA is flapping between `minReplicas` and `maxReplicas`, and a VPA-concepts authoring exercise that uses `kubectl apply --dry-run=client` for verification because the assignment deliberately does not install VPA.

## Prerequisites

Complete `exercises/pods/assignment-5` (Resources and QoS) so that `resources.requests`, `resources.limits`, QoS classes, and the relationship between requests and scheduling decisions are familiar. HPA cannot compute CPU utilization without requests, and in-place resize modifies those same fields on a running pod; both skills depend on assignment-5's content. Complete `exercises/pods/assignment-7` (Workload Controllers) because HPA targets Deployments (and optionally StatefulSets); the reconciliation model from that assignment carries over directly. Familiarity with `kubectl top` output is helpful but not required; the tutorial introduces it.

## Cluster Requirements

A multi-node kind cluster is the right environment so that HPA scale-out actually distributes pods across workers. metrics-server is required for every exercise; without it every HPA's TARGETS column stays at `<unknown>` and no scaling happens. See `docs/cluster-setup.md#multi-node-kind-cluster` for the cluster creation command and `docs/cluster-setup.md#metrics-server` for the metrics-server install with the `--kubelet-insecure-tls` flag that kind requires. No other extras (MetalLB, Calico, Gateway API CRDs) are needed.

## Estimated Time Commitment

Plan for 60 to 90 minutes on the tutorial; many steps include a wait for metrics-server's 15-second scrape interval or the HPA's 15-second default sync period. The 15 exercises together take four to six hours, weighted toward Levels 3 and 5 where diagnostic reasoning is the main work. Level 1 runs about 15 minutes each because each exercise waits one or two HPA sync cycles. Level 2 exercises (especially in-place resize) each take 10 to 20 minutes. Level 3 debugging averages 20 to 30 minutes per exercise because HPA failures are silent (the HPA still exists; it just does not scale). Level 4 runs 30 to 45 minutes each; multi-metric HPAs require carefully crafted `metrics` arrays, and combining HPA with in-place resize requires coordinating two changes on the same pod set. Level 5 runs 40 to 60 minutes per exercise. Keep an eye on the HPA's TARGETS column throughout; an HPA reporting `<unknown>` is not scaling and the wait time will not help.

## Scope Boundary and What Comes Next

This assignment covers the HPA controller, in-place pod resize, and VPA concepts at the level the CKA exam tests. It does not cover static resource management (setting requests and limits, QoS classes, LimitRanges, ResourceQuotas), which is `exercises/pods/assignment-5`. It does not cover the workload controllers that HPA scales (Deployments, StatefulSets), which are `exercises/pods/assignment-7` and `exercises/statefulsets/`. It does not cover custom metrics adapters or external metrics APIs at implementation depth; the concept is introduced in the tutorial, but writing an adapter is out of CKA scope. It does not cover cluster autoscaling (scaling the node pool), which depends on a cloud-provider integration not available in kind. It does not cover PodDisruptionBudget, which is adjacent to HPA in production but not on the CKA curriculum. KEDA, Karpenter, and other event-driven or fleet-management autoscalers are all out of scope. The application-layer side of "HPA not scaling" reappears in `exercises/troubleshooting/assignment-1` as part of the cross-domain troubleshooting practice.

## Key Takeaways After Completing This Assignment

By the time you finish all 15 exercises, you should be able to install and verify metrics-server with the kind-specific `--kubelet-insecure-tls` flag, confirm that `kubectl top nodes` and `kubectl top pods` work before any HPA is created, and understand that every HPA depends on metrics-server being healthy. You should know the full HPA spec surface: `apiVersion: autoscaling/v2` (not v1, not v2beta2), `spec.scaleTargetRef` with `apiVersion`, `kind`, and `name`, `spec.minReplicas` and `spec.maxReplicas`, and the `spec.metrics` array with `Resource` type entries targeting CPU or memory as either `AverageUtilization` or `AverageValue`. You should know that HPA's CPU-utilization calculation requires `resources.requests.cpu` to be set on the target's pod template, and that the TARGETS column in `kubectl get hpa` stays at `<unknown>` until metrics-server has scraped at least once (roughly 15 to 30 seconds). You should be able to tune `spec.behavior.scaleUp` and `spec.behavior.scaleDown` independently, using `stabilizationWindowSeconds` to prevent flapping and `policies` to cap per-period scale changes. You should know that in-place pod resize is GA in Kubernetes 1.33, that it uses `spec.containers[].resizePolicy` with `RestartPolicy: NotRequired` or `RestartContainer` per-resource, and that CPU and memory default to `NotRequired` which means resizing them does not restart the container. You should know the three VPA modes at a concept level (`Off`, `Initial`, `Auto`) and why VPA and HPA cannot target the same resource on the same workload without conflict. Finally, you should be able to diagnose a non-scaling HPA in under five minutes using `kubectl describe hpa` (the `Conditions` and `Events` blocks), `kubectl top pod -l app=<label>` to confirm metrics are visible, and the `scaleTargetRef` to confirm the right target is named.
